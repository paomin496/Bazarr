No payment gateway, escrow, or bank linking in this version. Buyers and sellers connect through in-app messaging (and optionally WhatsApp) and settle payment/delivery between themselves. This removes the need for a licensed payment aggregator, KYC/AML flows, and bank account linking entirely — the platform's role is discovery, listings, and connecting buyer to seller, with the owner available to mediate disputes manually.

  • POST /auth/mfa/enable, POST /auth/mfa/verify

Users

  • GET /users/me, PATCH /users/me
  • GET /users/:id (public profile)

  • GET /listings (query: q, category, minPrice, maxPrice, condition, sellerRating)
  • POST /listings, PATCH /listings/:id, DELETE /listings/:id
  • POST /listings/:id/images
  • POST /listings/:id/report

Contact Seller / Messaging

  • POST /listings/:id/message — buyer taps "Message Seller"; creates (or reuses) a conversation and sends the first message
  • GET /conversations — mine, as buyer or seller
  • GET /conversations/:id/messages
  • POST /conversations/:id/messages
  • PATCH /conversations/:id — seller marks seller_confirmed_sale once a deal is done off-platform (unlocks reviews)
  • Real-time delivery via Socket.io room per conversation; falls back to polling if socket unavailable

Disputes

  • POST /conversations/:id/disputes
  • GET /disputes/:id

Admin (role=admin guard on all)

  • GET /admin/dashboard/stats
  • GET/PATCH/DELETE /admin/users/:id, POST /admin/users/:id/suspend
  • GET/PATCH/DELETE /admin/listings/:id
  • GET /admin/disputes, POST /admin/disputes/:id/resolve
  • GET/PATCH /admin/cms/:slug

4.Key Flow — Message Seller

  1. Buyer views a listing, taps "Message Seller" → POST /listings/:id/message with an initial note (e.g. "Is this still available?").
  2. Backend creates a conversations row (or reuses the existing one for that buyer+listing) and the first messages row; seller gets a notification (in-app + email/SMS, optionally WhatsApp deep link).
  3. Buyer and seller negotiate price, payment, and delivery entirely in the chat thread — the platform does not process payment or hold funds.
  4. Once they've agreed and completed the deal (however they choose to pay/ship), the seller marks the conversation seller_confirmed_sale. This flips the listing to sold and unlocks mutual reviews.
  5. If something goes wrong, either party can raise a dispute on the conversation; the admin reviews the message log and mediates manually (no funds to release/refund since none were held).

This keeps the platform's role to discovery + trusted introduction, not transaction processing — much simpler to build and no financial licensing requirements.

7.Security & Non-Functional Checklist

  • Argon2/bcrypt password hashing; JWT + refresh tokens; MFA via TOTP (speakeasy/otplib)
  • Rate limiting (per-IP + per-user) on auth and listing-creation endpoints
  • Input validation (class-validator/zod), parameterized queries via Prisma (SQL-injection safe)
  • CSP, HSTS, SSL/TLS everywhere; secrets in a vault (AWS Secrets Manager), never in repo
  • Fraud flagging: rule-based first (price outliers, duplicate images/text, new-account+high-price), ML-assisted later
  • Testing: unit (Jest), integration (supertest against test DB), e2e (Playwright) covering the messaging and dispute flows
  • Audit log table for all admin actions (who did what, when)

8.What's Genuinely "Fill in Credentials" vs. Needs Real Setup

ItemStatus
Facebook OAuthNeeds a registered Meta app + App ID/Secret
Payments/escrow/bank linkingRemoved — buyers and sellers arrange payment and delivery themselves via chat
WhatsAppNo API required for a click-to-chat link; WhatsApp Business API only needed for automated notifications
Hosting/domain/SSLStandard AWS/GCP setup, certbot or ACM
Trade-off to keep in mind: without escrow, the platform can't guarantee a buyer gets their money back or a seller gets paid — it's closer to an OLX/Craigslist model than an eBay-with-buyer-protection model. Fraud prevention (listing flagging, reporting, ratings) matters more here since there's no financial safety net. Worth deciding early whether that trade-off is acceptable, or whether a lighter-weight option (e.g. just displaying a UPI ID for manual transfer, no platform involvement) is worth adding later without going all the way to escrow.
